PGSIMS FRONTEND - EXECUTIVE SUMMARY
================================================================================

OVERVIEW
================================================================================
The PGSIMS frontend is a Next.js (React) application that manages postgraduate
medical training systems. It provides role-based dashboards for residents,
supervisors, faculty, and administrators.

TECHNOLOGY STACK
================================================================================
- Framework:      Next.js 14+ (TypeScript)
- HTTP Client:    Axios
- State Mgmt:     Zustand (auth only) + React hooks
- UI Framework:   Tailwind CSS
- Component Lib:  Custom components + shadcn/ui (implicit)
- Testing:        Playwright (e2e)

API ARCHITECTURE
================================================================================
- Backend:        Django REST Framework (http://backend:8014)
- Auth:           JWT tokens (access + refresh)
- Proxy:          Next.js API route proxies requests to backend
- Interceptors:   Auto token refresh on 401
- Base URL:       Same-origin (/api/* → backend:/api/*)

KEY STATISTICS
================================================================================
- Total API Endpoints:        84+
- Service Modules:            11 (auth, training, userbase, notifications, bulk, audit, etc.)

API RESPONSE PATTERNS
================================================================================

Most endpoints return one of two formats:

1. Paginated: { count: number, results: T[] }
2. Array: T[]
3. Single object: T
4. File download: Blob (responseType: 'blob')

Helper function `toArray()` in training.ts handles both paginated and array
responses transparently.

AUTHENTICATION FLOW
================================================================================

1. User submits login form → POST /api/auth/login/
2. Backend returns { user, access, refresh }
3. Frontend stores in localStorage and Zustand store
4. Cookies synced for SSR compatibility
5. Authorization header added: "Bearer {access}"
6. On 401 response: POST /api/auth/refresh/ to get new access token
7. Retry original request with new token
8. On refresh failure: redirect to /login

TOKEN STORAGE
================================================================================

localStorage:
  - access_token        (JWT access token)
  - refresh_token       (JWT refresh token)
  - user                (JSON user object)
  - auth-storage        (Zustand persisted state)

Cookies (for SSR):
  - pgsims_access_token (access token)
  - pgsims_access_exp   (token expiry epoch)
  - pgsims_user_role    (user role)

ERROR HANDLING
================================================================================

- 401 responses: Auto token refresh
- 403 responses: Redirect to /unauthorized
- Network errors: User-friendly error messages
- Form validation: Client-side validation before submission
- API errors: Display backend error details when available

ENVIRONMENT CONFIGURATION
================================================================================

NEXT_PUBLIC_API_URL  - Client API base (optional, defaults to same-origin)
SERVER_API_URL       - Server API base (optional, defaults to http://backend:8014)
INTERNAL_API_URL     - API proxy backend (optional, defaults to http://backend:8014)

Note: In production, the frontend proxy (/api/*) routes requests to the Django
backend. This allows same-origin requests from the browser without CORS issues.

STATE MANAGEMENT
================================================================================

Auth State (Zustand):
  - Persisted to localStorage
  - Hydration check before rendering
  - Methods: setAuth(), setUser(), clearAuth()

UI State (React hooks):
  - Loading states
  - Form data
  - Error/success messages
  - Modal visibility

No Redux or Context API used - simple and lightweight.

PERFORMANCE CONSIDERATIONS
================================================================================

1. Lazy route-based code splitting (Next.js default)
2. Static rendering where possible
3. Client-side data fetching for dynamic content
4. Token refresh cached to avoid duplicate requests
5. Axios interceptors prevent unnecessary API calls

DEPLOYMENT
================================================================================

Frontend runs on:  http://localhost:3000 (dev) or production URL
Backend runs on:   http://backend:8014
API Proxy:         /api/[...path] → backend:/api/[...path]

The frontend uses Next.js as a proxy to avoid CORS issues and handle same-origin
API requests transparently.
